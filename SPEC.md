# Схематор

Схематор берёт пример JSON и помогает превратить его в чистую типовую схему на
языке **essensio**. Вы вставляете данные, даёте имена повторяющимся структурам —
и получаете готовое описание типов. Нотация следует грамматике
[essensio/notation](https://github.com/essensio/notation).

## Зачем он нужен

Когда у вас на руках только пример ответа какого-нибудь API, структура в нём есть,
но она безымянная. Глядя на массив заказов, человек понимает, что каждый его
элемент — это «Заказ», но в самом JSON такого имени нет. Схематор выводит структуру
автоматически и даёт удобно её назвать, чтобы из сырого примера получилась
переиспользуемая схема.

Возьмём такой ввод:

```json
{
  "orders": [
    { "address": "Машиностроителей 15", "date": "2024-04-23", "qty": 2 },
    { "address": "Ленина 1",            "date": "2024-04-24", "qty": 5 }
  ]
}
```

Сразу после вставки Схематор выводит типы, но все структуры пока безымянные:

```
{orders: {address: Строка, date: Строка, qty: Число}[]}
```

Элемент массива — `{address, date, qty}` — это **анонимный тип**, и Схематор его
подсвечивает. Вы кликаете по нему, вводите имя «Заказ» — и схема становится чистой:

```
Заказ = {address: Строка, date: Строка, qty: Число}

{orders: Заказ[]}
```

Теперь у структуры есть имя, на неё можно ссылаться, и описание читается как
настоящая схема, а не как распечатка примера.

## Анонимные типы

Анонимный тип — это безымянный **кортеж** (JSON-объект): на него нельзя сослаться,
пока он не назван. Именно кортежи Схематор подсвечивает и предлагает назвать.
Безымянный **союз** (разнородный массив) — тоже анонимный тип: его именуют целиком
или по членам (см. «Разнородный массив»). **Отношение** (массив) отдельным типом не
объявляют — его таблица в нотации неявна,
она подразумевается `#` элемента-сущности; поэтому сам массив не именуется, имя
получает его **элемент** (`{orders: Заказ[]}`, а не `Заказы = Заказ[]`). Скаляры —
число, строка, булево, `Пусто` — системные типы; их можно назвать подтипом-доменом
(`Количество = Число`), но в подсветке анонимности они не участвуют.

Назвать анонимный тип — единственное действие разметки. Назвали — он поднимается
наверх отдельной декларацией `Имя = …`, а на прежнем месте остаётся ссылка по
имени. Не назвали — он так и остаётся встроенным в описание и подсвеченным,
напоминая, что его стоит назвать.

## Как выводятся типы

Каждое значение JSON получает тип по простым правилам:

| Значение JSON | Тип essensio |
|---|---|
| `null` | `Пусто` |
| `true` / `false` | `Булево` |
| число | `Число` |
| строка | `Строка` |
| объект | кортеж `{поле: тип, …}` |
| массив объектов | отношение `Кортеж[]` (поля объединяются) |
| массив однотипных скаляров | `Число[]`, `Строка[]` и т. п. |
| разнородный массив | union элемента — `(Число \| Строка)[]`, `(T \| Пусто)[]` |

Строка остаётся `Строка`: Схематор не угадывает из неё `Дата` или другие сужения —
это отдельное решение о типе, а не разметка имени.

**Объединение полей.** Если в массиве лежат объекты с **разным набором полей**,
Схематор сводит их к одному кортежу, объединяя все поля: поле, что есть лишь в части
объектов, включается и берёт тип оттуда, где присутствует. Так `[{a:1,b:2},{a:3}]`
даёт `{a: Число, b: Число}[]`.

**Разнородный массив.** Если элементы массива разных видов — скаляры разных доменов,
объект вперемешку со скаляром, что-то вперемешку с `null` — Схематор сводит элемент к
**объединению** (union): `[1, "a"]` → `(Число | Строка)[]`, `[{…}, null]` →
`({…} | Пусто)[]`. Необязательность как раз и выражается членом `Пусто`. Порядок
членов канонический (кортеж, отношение, домены, `Пусто`), поэтому от порядка элементов
в массиве он не зависит.

Именовать в союзе можно двояко, и обе формы сочетаются. **Весь союз** — одним именем:
назвав `(Число | Строка)` «Значение», получаем `Значение = Число | Строка` и элемент
`Значение[]`. **Каждый член** — по отдельности: назвав члены «Целое» и «Текст»,
получаем `Целое = Число`, `Текст = Строка` и элемент `(Целое | Текст)[]`. То же с
кортежем-членом: назвав кортеж в `({…} | Пусто)` «Заказ», получаем `Заказ = {…}` и
элемент `(Заказ | Пусто)[]`. Имя союзу и имена членам Схематор предлагает у самого
массива, поэтому именование не зависит от порядка элементов. Отношение-член имени не
получает — его таблица неявна (см. «Анонимные типы»).

**Пустой массив** не даёт узнать тип элемента: Схематор печатает его отношением
пустого кортежа-заглушки — `{}[]`; кортеж обретёт поля, когда в данных появятся
элементы, и его можно назвать заранее.

## Похожие блоки и общее имя

Две структуры считаются одинаковыми, когда у них одна и та же форма — один и тот же
набор полей с одинаковыми типами. В примере выше оба заказа одной формы, поэтому
достаточно одного имени «Заказ» на обоих.

Схематор не объединяет одинаковые блоки сам и не именует их пакетно. По умолчанию имя
получает только тот блок, который вы назвали. Если хотите одно имя на нескольких
блоках одной формы — введите это имя на каждом из них вручную; Схематор такой ввод
принимает, потому что форма у блоков одна (см. ниже).

Одно имя описывает одну форму. Если имя уже занято блоком одной формы, его нельзя
дать блоку другой формы: Схематор не примет такой ввод — поле имени ведёт себя как
при недопустимом имени (имя не сохраняется, поле остаётся открытым для исправления).
Так имя всегда указывает ровно на одну форму, и схема не зависит от того, какой блок
вы назвали первым.

Занятость имени смотрится по форме, а не по месту. Дать одно имя двум блокам **одной**
формы можно — так одно имя ложится на несколько одинаковых блоков, и две разные формы
при желании можно назвать по-разному. Запрещён лишь случай, когда имя уже стоит на
блоке одной формы, а его пытаются дать блоку **другой** формы. Правило одинаково
работает и внутри союза: если назвать весь союз `(Число | Строка)` именем «Зн», то
этим же именем нельзя назвать его член `Строка` — у целого союза и у его члена формы
разные.

Проверка идёт **в момент ввода имени** — занятое имя другой формы просто не
принимается. Но имя живёт и после ввода: правка JSON может изменить форму уже
названного блока. Если после такой правки одно имя начинает накрывать две разные
формы, Схематор не печатает по нему неверную схему — разъехавшиеся имена он снимает
с печати (перестаёт применять, блок снова показан как неназванный). Поэтому конфликтного
состояния схемы не возникает ни от ввода, ни от правки текста: схема всегда строится
из непротиворечивого набора имён.

## Состояние и пересчёт

Всё, что вводит пользователь, — это текст JSON и присвоенные имена. Больше ничего не
хранится. Выведенные типы, список анонимных блоков, подсветка и текст схемы — всё
это пересчитывается из текста и имён при каждом изменении. Один источник истины,
поэтому рассинхрона между «данными» и «схемой» не бывает в принципе.

Пересчёт детерминирован: одни и те же ввод и имена всегда дают одну и ту же схему,
без обращения к часам, случайности или внешним данным.

## Краевые случаи

Схематор корректно обрабатывает: пустой ввод; невалидный JSON (показывает ошибку с
позицией, схему не строит); пустой объект `{}` — это кортеж из нуля полей, его тоже
можно назвать; пустой массив; разнородный массив (именуем целиком и по членам,
независимо от порядка элементов); глубокую вложенность; строковые
ключи вроде `{"order-id": 1}`; поле со значением `null` (тип `Пусто`); числа с
экспонентой; и попытку дать занятое имя блоку другой формы (ввод не принимается, см.
«Имя описывает форму: конфликт имён»).

## Имя описывает форму: конфликт имён

Это строгое описание правила из «Похожие блоки и общее имя». Оно опирается на уже
существующие сущности ядра — карту имён `Names` и сигнатуру формы `signatureOfSet`
(`src/core/signature.ts`) — и новых сущностей не вводит.

**Имена.** Состояние имён — это отображение `names : Path → Имя` (путь позиции →
присвоенное имя). Путь позиции — тот же, что использует эмиттер: путь узла
(`$.a`, `$[]`) для одиночного типа и всего союза, путь `путь|тег` для именуемого члена
союза. Имя занятой позиции — это значение `names[path]`.

**Форма позиции — из самого пути.** У каждого пути `p`, который можно назвать, есть
форма — каноническая сигнатура множества значений этой позиции:
`форма(p) = signatureOfSet(значения_в(p))`. **Значения позиции (`значения_в(p)`)
определяются по `p`**, тем же обходом ветвей, что и эмиттер:

- корень `$` — `[value]`;
- поле кортежа `путь.k` — `valuesForKey(объекты_в(путь), k)`;
- элемент отношения `путь[]` — `flattenArrays(массивы_в(путь))`;
- весь союз — тот же путь, что у позиции (`$`, `путь.k`, `путь[]`): все значения
  позиции, как у любого узла; форма союза — сигнатура всего множества, вида `(a|b|…)`;
- **член союза `путь|тег`** — **только его ветвь**: подмножество значений союза,
  отнесённое к этому тегу (`{}` → объекты, `[]` → массивы, имя домена → скаляры этого
  домена, `Пусто` → null). Форма члена считается по **его ветви**, а не по всему союзу,
  поэтому `форма($[])` (весь союз `(Число | Строка)`) и `форма($[]|Строка)` (член
  `Строка`) различны.

Форму позиции всегда берём из `path` через `значения_в(path)`, а не из отдельного
аргумента. Прежний контрол передавал в `submit` ещё и аргумент `signature`, но для
всего союза и его членов туда приходил путь (`$[]`, `$[]|тег`), а не сигнатура формы;
этот аргумент обслуживал группировку «похожих» и удалён вместе с ней (см. ниже).
Теперь подпись операции ввода — `submit(path, name)`.

Сигнатура канонична: она не зависит ни от порядка ключей в объектах, ни от порядка
элементов разнородного массива, ни от того, какие объекты слились в кортеж (см.
`signatureOfValue`/`signatureOfSet`). Две позиции одной формы дают равные сигнатуры;
позиции разной формы — различные.

**Инвариант непротиворечивости — свойство пары (текст, names).** Допустимо лишь такое
состояние, в котором каждое имя стоит на позициях ровно одной формы. Инвариант связан
не с одной точкой ввода, а с парой «разобранный текст + карта имён»: форма позиции
вычисляется из текста, поэтому при фиксированном `names` правка текста может его
нарушить.

```
для данного текста T и карты имён names, для любых путей p, q:
  names[p] = names[q]  ⇒  форма_T(p) = форма_T(q)
```

Иначе одно имя описывало бы две разные формы, и печать схемы выбрала бы тело
произвольно — по порядку обхода. Инвариант держится **в двух точках**. Квантор «для
любых путей p, q» пробегает **активные пути** (см. ниже), а не весь `names`.

**Активные пути и тотальность формы.** Карта `names` переживает правки текста, поэтому
в ней могут лежать пути, которых в текущем тексте уже нет, и пути с именем, снятым с печати.
Форма определена лишь там, где есть значения: для исчезнувшего из текста пути `q`
множество `значения_в(q)` пусто и `форма(q)` **не вычислима**. Чтобы предикат был
тотален, квантор инварианта и предусловия пробегает **только активные пути** — те `q`
текущего разбора, для которых одновременно:

- **путь присутствует в текущем разобранном тексте** — `значения_в(q)` непусто, поэтому
  `форма(q)` **вычислима** (исчезнувший из текста путь сюда не попадает: его форма не
  определена, и квантор его обходит); И
- **имя `names[q]` не снято с печати** на этом пути — то есть под текущим текстом оно стоит на
  путях ровно одной формы (определение снятия — «Точка 2»). Имя снимается с печати
  ровно тогда, когда оно накрыло несколько форм; такой путь из квантора занятости
  исключается, поэтому снятое с печати имя занятым **не считается**.

Так область квантора однозначна и тотальна: `форма(q)` берётся **лишь там, где путь `q`
реально присутствует в текущем разборе** (текст + значения), а снятые с печати и исчезнувшие
пути в проверке «занятости имени» не участвуют. Снятое с печати либо исчезнувшее из текста имя
позицию **не занимает**. Этим точки 1 и 2 не спорят: то, что схема не печатает
(призрачный путь или путь со снятым именем), не блокирует и ввод.

**Точка 1 — ввод (`submit`).** Имена меняются через присвоение `submit(path, name)`.
Присвоение `names[path] := N` **принимается** тогда и только тогда, когда:

- `isValidTypeName(N)` (имя допустимо грамматикой движка — действующая проверка); И
- имя `N` свободно для формы позиции `path`: не существует **активного** пути
  `q ≠ path` такого, что `names[q] = N` и `форма(q) ≠ форма(path)`.

Квантор по активным путям, не по всему `names`: `форма(q)` берётся лишь там, где путь `q`
присутствует в текущем разборе, поэтому имя, снятое с печати на пересчёте либо оставшееся в
`names` от исчезнувшего из текста пути, занятым не считается и ввод не блокирует (никакого
«призрака»). Иначе присвоение **отклоняется**: `names` не меняется,
поле остаётся открытым (поведение как при недопустимом имени). Это предусловие операции
ввода: корректное состояние строится, а не чинится. Групповой/пакетной ветви
(наименование «всех похожих» одним движением) **нет** — несколько блоков одной формы
называются повторным ручным вводом того же имени, и каждый такой ввод проходит ту же
проверку (форма совпадает → имя свободно → принято).

**Точка 2 — пересчёт из текста (`analyze`/`emit`).** Текст правится свободно, имена при
этом сохраняются, поэтому форма уже названной позиции может «съехать» так, что одно имя
начнёт накрывать две разные формы. На пересчёте схема **не печатается неверно**: имена,
у которых под текущим текстом нарушился инвариант, **снимаются с печати** — к таким позициям имя
не применяется (узел показан как неназванный, его тело печатается анонимно).
Снятие с печати — производное (выводится из текста и `names`, не хранится отдельно): для каждого
имени берутся только **существующие в тексте** пути (исчезнувшие из текста — форма
не вычислима — в расчёт не идут); если среди них встречается более одной формы, имя
накрывает несколько форм и снимается с печати на всех своих путях. `names` при этом не
переписывается — снимается с печати лишь применение имени; вернёте текст к
совместимой форме — имя снова применится.

Молчаливый выбор «первое тело по обходу выигрывает» в `emit.ts` тем самым становится
недостижимым и **убирается как часть модели**, а не как опциональная защита: эмиттер
по построению печатает только согласованные имена. Результат не зависит ни от порядка
ключей, ни от порядка обхода.

**Закрытые сценарии** (из приёмки):

1. *Одно имя на две формы в разных позициях* (кортеж против скаляра): пути ведут к
   разным сигнатурам — второе присвоение отклоняется на вводе; если разъехались
   правкой текста — снимаются с печати на пересчёте.
2. *Независимость от порядка ключей*: форма — каноническая сигнатура, порядок ключей
   и элементов на неё не влияет, поэтому и решение «свободно/занято», и снятие с печати
   от порядка не зависят.
3. *Коллизия внутри одного союза* (имя всего союза = имя члена): путь союза и путь
   члена (`путь|тег`) дают разные формы (`форма($[])` ≠ `форма($[]|тег)`), потому что
   форма члена считается по его ветви; присвоить члену имя, уже стоящее на целом союзе,
   нельзя.
4. *Стык ввод↔снятие с печати — снятое имя не занимает форму*: имя `N` стоит на пути `p`
   (форма `F1`); правка текста съезжает форму `p` так, что `N` накрывает несколько форм,
   и на пересчёте `N` снимается с печати (Точка 2) — `names[p]` остаётся, но имя к печати не
   применяется и в квантор занятости не входит. Тогда `submit(p', N)` на новой позиции
   `p'` другой формы `F2` (`F2 ≠ F1`) **обязан принять**: путь `p` со снятым именем не активен,
   занятым не считается, активного пути имени `N` иной формы, чем `форма(p')`, нет.
   Это прямое следствие области квантора по активным путям (см. «Активные пути»).

## Интерфейс

Экран — две панели.

Слева — **разметка**: ваш JSON, аккуратно отформатированный, с подсвеченными
анонимными блоками. Клик по блоку открывает поле ввода имени. Имя типа элемента
массива привязано к самому массиву, а не к первому элементу, поэтому доступно при любом
порядке элементов. Для союза массив показывает его члены и даёт назвать как любой
член, так и весь союз. Несколько блоков одной формы называются одним именем повторным
вводом этого имени на каждом — отдельной кнопки «ко всем похожим» нет. Если введённое
имя уже занято блоком другой формы, поле его не принимает (как при недопустимом имени)
и остаётся открытым — конфликт не сохраняется. Если после правки JSON форма уже
названного блока разъехалась с другой позицией того же имени, имя снимается с печати: блок снова
показан как неназванный (анонимный), пока текст не вернётся к совместимой форме.

Справа — **схема**: готовое описание essensio, только для чтения, моноширинным
шрифтом. Ещё не названные анонимные типы подсвечены здесь так же, как слева.

Сверху — кнопки «Пример», «Очистить» и «Копировать». Пустой экран показывает
подсказку и кнопку загрузки примера; невалидный JSON — сообщение об ошибке вместо
схемы.

Цвет работает осмысленно. Нейтральная гамма — `slate`, основное действие — `indigo`,
подсветка анонимного типа — мягкий жёлтый маркер (`amber`), читающийся как «здесь
тип без имени». Все размеры, отступы и цвета берутся из стандартных шкал Tailwind.

## Архитектура

Ядро (`src/core`) — чистый TypeScript без Vue: разбор JSON, вывод типов, сравнение
форм и генерация текста essensio. Оно покрыто тестами и ничего не знает про
интерфейс. Модель типов в ядре — это представление схемы essensio.

Интерфейс (`src/components`) — Vue 3. Он хранит только текст и имена, а всё
остальное получает из ядра через вычисляемые свойства.

Сборка — Vite с относительным базовым путём, чтобы собранная статика работала из
любой подпапки на хостинге демо.
